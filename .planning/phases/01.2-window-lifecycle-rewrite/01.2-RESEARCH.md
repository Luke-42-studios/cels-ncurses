# Phase 1.2: Window Lifecycle Rewrite - Research

**Researched:** 2026-03-01
**Domain:** CELS lifecycle hooks (CEL_Lifecycle, CEL_Observe), composition patterns, component mutation
**Confidence:** HIGH

## Summary

This phase replaces the ECS bridge file (`ncurses_ecs_bridge.c`) and raw flecs observers with the new CEL_Lifecycle + CEL_Observe pattern. The research investigated the actual CELS APIs by reading the source code directly -- CEL_Lifecycle, CEL_Observe, component mutation, and composition lifecycle attachment.

The critical findings are:
1. **Lifecycle attachment works via the `.lifecycle` prop on compositions.** There is NO `cel_lifecycle()` composition body verb. The `_cel_compose_impl` macro checks `props.lifecycle` and calls `cels_lifecycle_push/pop` automatically. The NCursesWindow composition must either hardcode the lifecycle in its factory or pass it through props.
2. **There is no public CELS API for setting component data on an arbitrary entity outside of composition context.** `cels_component_add()` operates on the "current entity" from the composition stack. The `ncurses_component_set()` bridge function wraps `ecs_set_id()` directly. A replacement strategy is needed.
3. **CEL_Observe and CEL_Lifecycle are file-scope declarations** that auto-register via `__attribute__((constructor))`. They are the correct replacement for the raw flecs observers in the bridge.

**Primary recommendation:** Define `NCursesWindowLC` lifecycle and observers in `tui_window.c`. The NCursesWindow composition hardcodes the lifecycle internally (not exposed to consumer). Replace `ncurses_component_set()` with direct `ecs_set_id()` calls via `cels_get_world()` + `cels_component_notify_change()` for reactivity, keeping the flecs include isolated to `tui_window.c` through the CELS public `cels_get_world()` API.

## Standard Stack

### Core

| Library/API | Source | Purpose | Why Standard |
|-------------|--------|---------|--------------|
| `CEL_Lifecycle(Name, .active = &Cond)` | cels.h L585-592 | Define lifecycle with condition | Only API for file-scope lifecycle definitions |
| `CEL_Observe(Name, event) { body }` | cels.h L612-613 | File-scope event observer | Auto-registration via constructor attribute |
| `CEL_Condition(Name, States...) { body }` | cels.h L556-558 | Condition for lifecycle transitions | Standard boolean condition pattern |
| `cels_lifecycle_register_def()` | cels_runtime.h L98 | Register lifecycle from def pointer | Used by cel_init macro for .lifecycle prop |
| `cels_lifecycle_push/pop()` | cels_runtime.h L107-110 | Composition stack lifecycle binding | Automatic in cel_init when .lifecycle is set |
| `cels_component_notify_change()` | cels_api.c L518-528 | Trigger reactivity for cel_watch | Required for cel_watch(entity, Type) to work |
| `cels_get_world()` | cels_runtime.h L346 | Access underlying flecs world | Public API -- allows ecs_set_id without flecs include |
| `cels_resolve_observers()` | cels_runtime.h L121 | Resolve pending observer registrations | Called during session start automatically |

### Supporting

| API | Source | Purpose | When to Use |
|-----|--------|---------|-------------|
| `cels_lifecycle_bind_entity()` | cels_runtime.h L104 | Manual entity-lifecycle binding | When lifecycle not passed via .lifecycle prop |
| `cels_request_quit()` | cels_runtime.h L301 | Signal application quit | Already used in input system |
| `cels_entity_t entity` (observer body) | cels_macros.h L172 | Entity passed to CEL_Observe body | Access entity in observer callbacks |

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| Hardcode lifecycle in factory | Expose .lifecycle prop to consumer | Consumer shouldn't need to know about lifecycle internals |
| `ecs_set_id()` via `cels_get_world()` | New public `cels_entity_set_component()` API | Would require CELS repo change -- premature abstraction |
| Static global `g_window_entity` | ECS query from system | Query adds overhead, static is simpler for single-window |

### Pattern 1: Lifecycle + Observer for Terminal Init/Shutdown

**What:** CEL_Lifecycle defines the window active/suspended state. CEL_Observe hooks fire ncurses terminal init on create and shutdown on destroy.

**When to use:** When an entity's creation/destruction triggers side effects (terminal init, resource allocation).

**Example (from test_lifecycle_fsm.c and cels.h):**
```c
// tui_window.c

// --- Condition: window is active when ncurses is initialized ---
static bool g_ncurses_active = false;

CEL_Condition(NCursesActive) {
    return g_ncurses_active;
}

CEL_Lifecycle(NCursesWindowLC, .active = &NCursesActive);

// --- Observer: on_create initializes ncurses terminal ---
CEL_Observe(NCursesWindowLC, on_create) {
    // entity is available as cels_entity_t
    // Read NCurses_WindowConfig from the entity via flecs
    // Call ncurses_terminal_init(&config)
    // Set initial NCurses_WindowState on entity
    g_window_entity = entity;
}

// --- Observer: on_destroy shuts down ncurses terminal ---
CEL_Observe(NCursesWindowLC, on_destroy) {
    ncurses_terminal_shutdown();
    g_window_entity = 0;
}
```

**Key insight from CELS tests:** The `entity` variable is injected into the CEL_Observe body automatically. The observer fires synchronously during entity creation (`cels_lifecycle_manager_bind_entity` dispatches on_create). Components are readable from within observers via the flecs world.

### Pattern 2: Lifecycle Attachment via Composition Factory

**What:** The NCursesWindow composition internally hardcodes the lifecycle, so consumers don't need to pass `.lifecycle = NCursesWindowLC`.

**When to use:** When a module owns the lifecycle and consumers should not be aware of it.

**Example:**
```c
// ncurses_module.c

CEL_Compose(NCursesWindow) {
    // Option A: Lifecycle already pushed by cel_init if consumer passes .lifecycle
    // Option B: Module hardcodes lifecycle by calling bind directly

    cel_has(NCurses_WindowConfig,
        .title = cel.title,
        .fps = cel.fps,
        .color_mode = cel.color_mode
    );
    // NCurses_WindowState is set by the on_create observer, not here
}
```

**Critical detail:** The `_cel_compose_impl` macro in cels_macros.h (line 56-79) shows that if `.lifecycle` is set in props, it calls `cels_lifecycle_push(cels_lifecycle_register_def(props.lifecycle))` before the factory and `cels_lifecycle_pop()` after. This means:
- **Option A:** Add `.lifecycle` to NCursesWindow props so the factory always passes it.
- **Option B:** The composition body calls `cels_lifecycle_bind_entity()` manually.

The best approach: Have the module init register the lifecycle, and the composition body call `cels_lifecycle_bind_entity()` with the lifecycle ID directly. This keeps the lifecycle internal to the module.

### Pattern 3: Component Mutation from System Context

**What:** The frame system needs to update `NCurses_WindowState` on the window entity each frame. No composition context exists during system execution.

**When to use:** Whenever a system needs to mutate a component on a specific entity.

**Current bridge approach:**
```c
// ncurses_ecs_bridge.c (BEING DELETED)
void ncurses_component_set(cels_entity_t entity, cels_entity_t component_id,
                           const void* data, size_t size) {
    ecs_world_t* world = cels_get_world(cels_get_context());
    ecs_set_id(world, (ecs_entity_t)entity, (ecs_id_t)component_id, size, data);
}
```

**Replacement approach:** Move the same logic into `tui_window.c`:
```c
// tui_window.c - internal helper
static void ncurses_set_window_state(NCurses_WindowState* state) {
    ecs_world_t* world = cels_get_world(cels_get_context());
    ecs_set_id(world, (ecs_entity_t)g_window_entity,
               (ecs_id_t)NCurses_WindowState_id,
               sizeof(NCurses_WindowState), state);
    cels_component_notify_change(NCurses_WindowState_id);
}
```

**Key:** `cels_get_world()` is a public CELS API (declared in `cels_runtime.h`). Using it does NOT require `#include <flecs.h>` -- but calling `ecs_set_id()` DOES. The success criteria say "No `#include <flecs.h>` in any cels-ncurses source file." This creates a tension.

### Pattern 4: Resolving the flecs.h Constraint

The success criteria state "No `#include <flecs.h>` in any cels-ncurses source file." However, `ecs_set_id()` is a flecs API. Options:

**Option A: Forward-declare ecs_set_id (fragile, not recommended)**
**Option B: Add a `cels_entity_set_component()` public API to CELS (clean but requires CELS change)**
**Option C: Accept that `tui_window.c` includes `<flecs.h>` temporarily through `cels_get_world()` return type**

Analysis: `cels_get_world()` already returns `struct ecs_world_t*` (forward-declared in `cels_runtime.h` line 344). The return type doesn't require `<flecs.h>`. But calling ANY flecs function (`ecs_set_id`, `ecs_get_id`) does require the flecs header.

**Recommended resolution:** The cleanest path is to either:
1. Add a thin wrapper to CELS: `cels_entity_set_component(entity, component_id, data, size)` that delegates to `ecs_set_id` internally, OR
2. Use `cels_component_add` by temporarily setting the composition engine's current entity (hacky), OR
3. Accept `<flecs.h>` in `tui_window.c` as a temporary bridge, to be resolved when CELS adds the proper API

Given success criterion 7 explicitly forbids `#include <flecs.h>`, option 1 is the correct path. This requires a small CELS change: adding `cels_entity_set_component()` to the public API.

### Anti-Patterns to Avoid

- **Setting component data in observers:** The on_create observer should NOT set NCurses_WindowState via `cel_has()` because observers don't run in composition context. Use the entity-targeted set function.
- **Exposing lifecycle to consumers:** NCursesWindowLC should be internal to the module. Consumers should not need to write `NCursesWindow(.lifecycle = NCursesWindowLC)`.
- **Running ncurses init outside observer context:** Terminal init must happen in the on_create observer so it's properly sequenced with entity creation and lifecycle binding.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Observer auto-registration | Manual function calls in module init | CEL_Observe macro with constructor attribute | Auto-registration is part of CELS design; manual registration is fragile |
| Lifecycle state machine | Custom WindowState enum | CEL_Lifecycle + CEL_Condition | CELS FSM handles active/suspended/destroyed with cascade |
| Component reactivity | Manual dirty tracking for resize | `cels_component_notify_change()` + `cel_watch()` | CELS reactivity system already handles dirty tracking |
| Entity lifecycle binding | Manual lifecycle push/pop in composition | `.lifecycle` prop or `cels_lifecycle_bind_entity()` | CELS composition engine handles push/pop automatically when props have lifecycle |

**Key insight:** The CELS lifecycle system already provides everything the old ECS bridge did: event-driven init/shutdown, state tracking, and reactivity. The bridge was a workaround for the absence of this API.

## Common Pitfalls

### Pitfall 1: Observer Body is NOT Composition Context

**What goes wrong:** Calling `cel_has()` inside a CEL_Observe body to add components to the entity. `cel_has()` uses `cels_component_add()` which targets the "current entity" from the composition stack -- but observers fire during `cels_lifecycle_manager_bind_entity()`, not during composition.

**Why it happens:** The CEL_Observe documentation says "All composition body capabilities are available" but this refers to reading (cel_read, cel_watch), not necessarily writing (cel_has targets composition stack entity, not observer entity).

**How to avoid:** Use `ecs_set_id()` via `cels_get_world()` or the proposed `cels_entity_set_component()` to set components on the `entity` parameter directly.

**Warning signs:** Component data appears on wrong entity or not at all.

### Pitfall 2: NCursesWindowLC Registration Timing

**What goes wrong:** CEL_Lifecycle defines a file-scope static. If it's defined in `tui_window.c` but CEL_Compose(NCursesWindow) in `ncurses_module.c` needs the lifecycle ID, there's a cross-TU visibility issue.

**Why it happens:** `CEL_Lifecycle(NCursesWindowLC)` creates `static cels_lifecycle_def_t*` -- static linkage means it's only visible in its translation unit.

**How to avoid:** Either:
- Define the lifecycle in the same TU as the composition (ncurses_module.c), OR
- Expose the lifecycle def pointer via an extern, OR
- Use `cels_lifecycle_bind_entity()` in the observer body instead of relying on composition-level lifecycle attachment

**Best approach:** Define `CEL_Lifecycle(NCursesWindowLC)` in `tui_window.c` and have the on_create observer call `cels_lifecycle_bind_entity()`. The composition in ncurses_module.c does NOT attach lifecycle -- the observer self-binds.

**WAIT -- re-analysis:** Actually, looking at the `_cel_compose_impl` macro more carefully, the lifecycle is pushed BEFORE the factory runs. The observer fires during `cels_end_entity()` -> `cels_lifecycle_manager_bind_entity()`. But `cels_lifecycle_push()` already happened. So entities created within the composition scope are automatically bound to the pushed lifecycle. The on_create observer fires as part of `cels_lifecycle_manager_bind_entity()`.

So the correct flow is:
1. Consumer writes `NCursesWindow(.title = "X") {}`
2. cel_init expands: if `.lifecycle` is set, push lifecycle
3. Factory runs `NCursesWindow_impl()` which does `cel_has(NCurses_WindowConfig, ...)`
4. After factory, `cels_end_entity()` commits the entity
5. If lifecycle was pushed, the entity is bound to it -> on_create fires
6. on_create observer receives entity, reads config, inits ncurses, sets window state

But the NCursesWindow composition needs to push the lifecycle automatically without the consumer specifying it. **This means the composition factory itself should push the lifecycle.** Since CEL_Lifecycle is static-linkage, the lifecycle must be defined in the same TU as CEL_Compose, OR the factory must call `cels_lifecycle_bind_entity()` directly after entity creation.

### Pitfall 3: Component IDs Not Yet Registered in Observer

**What goes wrong:** The on_create observer tries to use `NCurses_WindowConfig_id` to read the config from the entity, but the component ID might be 0 (not yet registered in this TU).

**Why it happens:** `CEL_Component(NCurses_WindowConfig)` creates a `static cels_entity_t NCurses_WindowConfig_id = 0` per translation unit. If `cel_has()` hasn't been called in this TU yet, the ID isn't populated.

**How to avoid:** Call `cels_ensure_component()` at the start of the observer body, or ensure the module init registers all component IDs before observers fire.

**Current code already handles this:** The module init in ncurses_module.c calls `cels_ensure_component(&NCurses_WindowConfig_id, ...)`. Since the observer fires after module init (entity creation happens in the composition body which runs after cels_register), the IDs are already populated.

**But cross-TU issue:** If the lifecycle and observer are in `tui_window.c`, the `NCurses_WindowConfig_id` in that TU is a different static variable than the one in `ncurses_module.c`. The observer in tui_window.c has its OWN `NCurses_WindowConfig_id` which will be 0.

**Solution:** The observer must call `cels_ensure_component()` for any component IDs it uses, OR the module init must share component IDs across TUs via extern linkage.

### Pitfall 4: CEL_Observe Constructor Runs Before main()

**What goes wrong:** The `__attribute__((constructor))` on CEL_Observe fires at program startup, before CELS is initialized. The observer just registers itself in a linked list -- no CELS APIs are called. The actual resolution happens in `cels_resolve_observers()` during session start.

**Why it happens:** This is by design. The constructor adds to a linked list; actual lifecycle resolution is lazy.

**How to avoid:** Just be aware -- don't call CELS APIs in the constructor. The body only runs when the lifecycle event fires.

### Pitfall 5: draw_test Target Does Not Use CELS

**What goes wrong:** The `draw_test` target compiles `tui_frame.c` with `#ifdef CELS_HAS_ECS` -- only the CELS-enabled path is active for the `minimal` target. The `draw_test` target doesn't define `CELS_HAS_ECS` and doesn't link CELS.

**Why it happens:** draw_test is a standalone ncurses test that doesn't use the CELS framework.

**How to avoid:** Ensure `tui_window.c` changes don't break the non-CELS code path. The lifecycle/observer code should be behind `#ifdef CELS_HAS_ECS` guards, or tui_window.c should not be included in the draw_test target (currently it's not listed in draw_test sources, only in INTERFACE sources).

**Current state:** Looking at CMakeLists.txt, `draw_test` doesn't include `tui_window.c` in its source list. It only includes graphics, layer, and frame files. So tui_window.c changes are safe for draw_test.

## Code Examples

### Example 1: Full NCursesWindowLC Lifecycle Definition

```c
// tui_window.c

#include <cels/cels.h>
#include <cels-ncurses/tui_ncurses.h>
#include <ncurses.h>  // STILL NEEDED for ncurses calls (initscr, etc.)

// Window active condition -- true when ncurses terminal is initialized
CEL_Condition(NCursesActive) {
    return g_ncurses_active;  // internal static flag
}

// Lifecycle: active when terminal is up
CEL_Lifecycle(NCursesWindowLC, .active = &NCursesActive);

// on_create: initialize ncurses terminal
CEL_Observe(NCursesWindowLC, on_create) {
    if (g_window_entity != 0) {
        fprintf(stderr, "[NCurses] Warning: only one window entity allowed\n");
        return;
    }
    g_window_entity = entity;

    // Read config from entity
    // Need cels_ensure_component for cross-TU ID resolution
    cels_ensure_component(&NCurses_WindowConfig_id, "NCurses_WindowConfig",
                          sizeof(NCurses_WindowConfig), CELS_ALIGNOF(NCurses_WindowConfig));
    cels_ensure_component(&NCurses_WindowState_id, "NCurses_WindowState",
                          sizeof(NCurses_WindowState), CELS_ALIGNOF(NCurses_WindowState));

    // Read config via ecs_get_id (requires cels_get_world)
    ecs_world_t* world = cels_get_world(cels_get_context());
    const NCurses_WindowConfig* config =
        ecs_get_id(world, entity, NCurses_WindowConfig_id);

    ncurses_terminal_init((NCurses_WindowConfig*)config);

    // Set initial WindowState
    NCurses_WindowState state = {
        .width = COLS, .height = LINES, .running = true,
        .actual_fps = (float)config->fps,
        .delta_time = 1.0f / (config->fps > 0 ? (float)config->fps : 60.0f)
    };
    ecs_set_id(world, entity, NCurses_WindowState_id,
               sizeof(NCurses_WindowState), &state);
    cels_component_notify_change(NCurses_WindowState_id);
}

// on_destroy: shut down ncurses terminal
CEL_Observe(NCursesWindowLC, on_destroy) {
    ncurses_terminal_shutdown();
    g_window_entity = 0;
}
```

**NOTE:** This example still uses `ecs_set_id()` and `ecs_get_id()` which require `#include <flecs.h>`. If success criterion 7 is strict, a CELS wrapper API is needed.

## Open Questions

### 1. flecs.h Include Constraint (CRITICAL)

**What we know:** Success criterion 7 says "No `#include <flecs.h>` in any cels-ncurses source file." However, `ecs_set_id()` and `ecs_get_id()` require `<flecs.h>`. The public CELS API `cels_get_world()` returns `struct ecs_world_t*` (forward-declared) but calling any flecs function requires the full header.

**What's unclear:** Whether the constraint means:
- (A) Literally no `#include <flecs.h>` line in any source file -- requires a new CELS public API, OR
- (B) No raw flecs observer/component-set boilerplate -- the spirit of the constraint is "use CELS abstractions"

**Recommendation:** Interpret as (A) -- literal. Add a small public API to CELS:
```c
// In cels_runtime.h:
extern void cels_entity_set_component(cels_entity_t entity, cels_entity_t component_id,
                                       const void* data, size_t size);
extern const void* cels_entity_get_component(cels_entity_t entity, cels_entity_t component_id);
```
This is a 10-line addition to CELS that wraps `ecs_set_id`/`ecs_get_id`. It eliminates the need for `<flecs.h>` in any cels-ncurses file. This should be a prerequisite task in the plan.

**If CELS change is not desired:** Accept `<flecs.h>` in `tui_window.c` via `cels_get_world()`, but note this violates success criterion 7.

### 2. Lifecycle Cross-TU Visibility

**What we know:** `CEL_Lifecycle` creates `static` variables. The composition (ncurses_module.c) and the lifecycle/observers (tui_window.c) are different TUs.

**What's unclear:** The best way to connect them.

**Recommendation:** Use one of these approaches:
- **(Preferred)** Define `NCursesWindowLC` with extern linkage in tui_window.c, declare it extern in tui_internal.h. The composition factory passes it via `.lifecycle` prop in the cel_init expansion.
- **(Alternative)** Don't use `.lifecycle` prop at all. Have the on_create observer be a generic hook registered via module init. The composition uses `cels_lifecycle_bind_entity()` after entity creation.
- **(Simplest)** Move lifecycle and observers into ncurses_module.c. Condition function and terminal init/shutdown are called via function pointers from tui_window.c.

### 3. Observer Firing During Entity Creation vs After

**What we know:** From test_lifecycle_fsm.c (test 18), the on_create observer fires during `cels_lifecycle_manager_bind_entity()` which happens in `cels_end_entity()`. At this point, the entity exists and has its components (set by cel_has during factory).

**What's unclear:** Whether `NCurses_WindowConfig` is readable from the entity within the on_create observer body. Based on the code flow:
1. `cels_begin_entity("NCursesWindow")` -- entity created
