**include/cels-ncurses/ncurses.h** -- New umbrella header (replaces backend.h). Includes:
- `<cels-ncurses/tui_ncurses.h>` (component types + module + compositions)
- `<cels-ncurses/tui_types.h>` (coordinate types, unchanged)

- Does NOT include tui_engine.h or backend.h (those are legacy)
- Does NOT re-export `<cels/cels.h>` -- developer includes that separately

**include/cels-ncurses/tui_ncurses.h** -- Public component types + module declaration:
- `#include <cels/cels.h>` (for CEL_Component, CEL_Define, cels_entity_t)
- `#include <stdbool.h>`
- `extern cels_entity_t NCurses;` and `extern void NCurses_init(void);` (module forward decls)
- `static inline void NCurses_register(void) { NCurses_init(); }` -- This is REQUIRED because `cels_register(NCurses)` expands to `NCurses_register()`, but the `CEL_Module(NCurses)` macro in ncurses_module.c generates `NCurses_register` as `static inline` (visible only in that TU). Consumer TUs like minimal.c need their own per-TU definition that delegates to the externally-linked `NCurses_init()`. This mirrors the per-TU register pattern used by `CEL_Component`.
- `CEL_Component(NCurses_WindowConfig)` with fields: `const char* title; int fps; int color_mode;`
- `CEL_Component(NCurses_WindowState)` with fields: `int width; int height; bool running; float actual_fps; float delta_time;`
- `CEL_Define(NCursesWindow, const char* title; int fps; int color_mode;);` -- public composition
- `#define NCursesWindow(...) cel_init(NCursesWindow, __VA_ARGS__)` -- call macro for natural syntax
- Do NOT include NCurses_InputState here yet (that's Phase 3). Do NOT forward-declare NCurses_Layer yet (Phase 4).
- Use standard include guards: `CELS_NCURSES_TUI_NCURSES_H`

**include/cels-ncurses/tui_internal.h** -- Internal function declarations:
- `#ifndef CELS_NCURSES_TUI_INTERNAL_H` guard
- Forward declarations for registration functions that ncurses_module.c calls:
  - `extern void ncurses_register_window_observer(void);`
  - `extern void ncurses_register_window_remove_observer(void);`
  - `extern void ncurses_register_input_system(void);`
  - `extern void ncurses_register_frame_systems(void);`
- These are implemented in tui_window.c (Plan 02), tui_input.c, tui_frame.c
- This header is NOT included by consumers -- only by ncurses_module.c

Note: The CEL_Component macro generates `static cels_entity_t NCurses_WindowConfig_id = 0;` per TU. This is fine -- `cels_ensure_component` deduplicates by name string. This is the standard CELS INTERFACE library pattern.

**src/ncurses_module.c** -- New module definition file:
- `#include "cels-ncurses/tui_ncurses.h"` (for component types)
- `#include "cels-ncurses/tui_internal.h"` (for registration function decls)
- `#include <flecs.h>` (for `cels_ensure_component`)
- `CEL_Module(NCurses) { ... }` body that:
  1. Registers NCurses_WindowConfig component via `cels_ensure_component(&NCurses_WindowConfig_id, "NCurses_WindowConfig", sizeof(NCurses_WindowConfig), CELS_ALIGNOF(NCurses_WindowConfig));`
  2. Registers NCurses_WindowState component via same pattern
  3. Calls `ncurses_register_window_observer();` (stub for now -- Plan 02 implements)
  4. Calls `ncurses_register_window_remove_observer();` (stub for now)
  5. Calls `ncurses_register_input_system();` (existing code will be refactored in Phase 3, stub here)
  6. Calls `ncurses_register_frame_systems();` (existing code, will be called from here)
- `CEL_Compose(NCursesWindow) { ... }` body that:
  1. Receives `cel` with fields `.title`, `.fps`, `.color_mode` (from CEL_Define props struct)
  2. Calls `cel_has(NCurses_WindowConfig, .title = cel.title, .fps = cel.fps, .color_mode = cel.color_mode);`
  3. That's it -- the observer (Plan 02) reacts to NCurses_WindowConfig being set

IMPORTANT: The `CEL_Module(NCurses)` macro expands to define `cels_entity_t NCurses = 0;` at file scope. This definition must exist in exactly ONE .c file (ncurses_module.c). The public header uses `extern` only.

Also create STUB implementations for the registration functions that don't exist yet, so the module compiles before Plan 02 wires them. Add these at the bottom of ncurses_module.c with `__attribute__((weak))` or just as plain stubs with a comment that Plan 02 replaces them:
```c
/* Stubs -- replaced by real implementations in Plan 02 (tui_window.c) */
void ncurses_register_window_observer(void) {}
void ncurses_register_window_remove_observer(void) {}
/* Input stub -- replaced in Phase 3 */
void ncurses_register_input_system(void) {}
/* Frame systems -- already exist in tui_frame.c, just need to be called */
/* Note: tui_frame_register_systems() already exists but with a different name.
   For now, stub it. Plan 02 will wire tui_frame.c's existing function. */
void ncurses_register_frame_systems(void) {}
```

Use `__attribute__((weak))` on all stubs so that when tui_window.c provides the real implementation in Plan 02, the linker picks the real one. This avoids multiple definition errors in the INTERFACE library pattern.
