# Phase 1.1: CELS API Purge - Research

**Researched:** 2026-02-28
**Domain:** CELS abstraction layer / flecs ECS API elimination
**Confidence:** HIGH

## Summary

This phase eliminates all direct flecs/ECS API usage from cels-ncurses source files, replacing them with CELS abstractions. Research was conducted by auditing every `.c` and `.h` file in cels-ncurses for raw flecs API calls, then systematically checking the CELS library (`/home/cachy/workspaces/libs/cels`) for corresponding abstractions.

Three source files contain raw flecs API calls: `tui_window.c` (observer registration + component set), `tui_input.c` (system registration), and `tui_frame.c` (system registration). The remaining files (graphics, layer, ncurses_module.c) are already clean. The CELS library provides `CEL_System` with `cel_run` for no-query systems, `cels_ensure_component` for component registration, and custom phase support via `cel_phase`. However, CELS does **not** provide a public abstraction for ECS-level observers (OnSet/OnRemove event callbacks) -- this is the primary gap requiring a thin wrapper.

**Primary recommendation:** Replace all 3 system registrations with `CEL_System(.phase = X) { cel_run { ... } }`, create a thin `cels_observer_register()` helper for observer registration (CELS lacks this abstraction), and replace all `ecs_set_id()` calls with a thin `cels_component_set_on()` helper. The `#include <flecs.h>` in all 3 files can then be removed entirely.

## Architecture Patterns

### Pattern 1: CEL_System with cel_run (No-Query Standalone System)

**What:** Replace raw `ecs_system_init()` + `ecs_iter_t*` callback with the CELS `CEL_System` macro. For systems that don't query components (like input reading or frame begin/end), use `cel_run` which skips the registration pass.

**When to use:** Any system that runs once per frame at a specific pipeline phase but doesn't iterate over entities. All 3 current cels-ncurses systems are this pattern.

**Example -- current (raw flecs):**
```c
// tui_input.c -- current code
#include <flecs.h>

static void tui_input_system_callback(ecs_iter_t* it) {
    (void)it;
    tui_read_input_ncurses();
}

void ncurses_register_input_system(void) {
    ecs_world_t* world = cels_get_world(cels_get_context());
    ecs_system_desc_t sys_desc = {0};
    ecs_entity_desc_t entity_desc = {0};
    entity_desc.name = "NCurses_InputSystem";
    ecs_id_t phase_ids[3] = {
        ecs_pair(EcsDependsOn, EcsOnLoad),
        EcsOnLoad,
        0
    };
    entity_desc.add = phase_ids;
    sys_desc.entity = ecs_entity_init(world, &entity_desc);
    sys_desc.callback = tui_input_system_callback;
    ecs_system_init(world, &sys_desc);
}
```

**Example -- after (CELS abstraction):**
```c
// tui_input.c -- after purge
// No #include <flecs.h> needed

CEL_System(NCurses_InputSystem, .phase = OnLoad) {
    cel_run {
        tui_read_input_ncurses();
    }
}

void ncurses_register_input_system(void) {
    /* Running pointer bridge + key/mouse init (unchanged) */
    g_tui_running_ptr = tui_window_get_running_ptr();
    define_key("\033[1;5A", CELS_KEY_CTRL_UP);
    /* ... etc ... */

    NCurses_InputSystem_register();
}
```

**Key insight:** `CEL_System` generates a `Name_register()` function. The existing `ncurses_register_*` functions become thin wrappers that do pre-registration setup then call `Name_register()`.

### Pattern 3: Thin Wrappers for CELS Gaps

**What:** Two flecs operations used in cels-ncurses have no direct CELS public API equivalent. Rather than `#include <flecs.h>` behind `#ifdef` guards, create thin helper functions in a new internal utility file.

**Gap 1: Observer Registration**

CELS has no public macro or function for registering ECS-level observers (OnSet/OnRemove event callbacks). Internally CELS uses raw `ecs_observer_init()` in its own `.c` files (`composition_engine.c`, `world_manager.c`), but this is not exposed.

**Solution:** Create a thin wrapper function that takes CELS-level types and internally calls flecs:

```c
// ncurses_ecs_bridge.c (new file, single point of flecs contact)
#include <cels/cels.h>
#include <flecs.h>   // THE ONLY file that includes flecs.h

typedef void (*ncurses_observer_callback_t)(cels_entity_t* entities,
                                             int count,
                                             void* component_data,
                                             cels_entity_t world_entity);

void ncurses_observer_register(const char* name,
                                cels_entity_t component_id,
                                int event,  // 0=OnSet, 1=OnRemove
                                void (*raw_callback)(void*));
```

Alternative (simpler): Since the observer callbacks need access to iterator fields (`it->entities`, `it->count`, `it->world`), and CEL_System's `cel_run` pattern doesn't support event observers, the simplest approach is a single bridge file that keeps `#include <flecs.h>` and exposes clean internal functions to the rest of the module.

**Gap 2: Component Set on Arbitrary Entity**

The CELS public API only allows setting components during composition (via `cel_has` -> `cels_component_add` which operates on the "current entity" from the composition engine). The cels-ncurses window code needs to set `NCurses_WindowState` on a specific entity ID outside composition context (in observers and frame update).

`cels_component_registry_add(reg, entity, comp_id, data, size)` exists internally but is not public. It calls `ecs_set_id()` underneath.

**Solution:** A thin wrapper that calls through the CELS internals:

```c
// ncurses_ecs_bridge.c
void ncurses_component_set(cels_entity_t entity,
                            cels_entity_t component_id,
                            const void* data,
                            size_t size);
```

### Anti-Patterns to Avoid

- **Scattering `#ifdef` guards for flecs.h:** Do NOT put `#ifdef CELS_LACKS_OBSERVER` guards in multiple files. Concentrate all flecs contact in ONE bridge file.
- **Reimplementing CEL_System manually:** The `CEL_System` macro generates all the boilerplate. Do not hand-write `ecs_system_desc_t` code when `CEL_System(.phase = X) { cel_run { body } }` works.
- **Using cels_get_world() throughout:** After the purge, only the bridge file should call `cels_get_world()`. Other files should call bridge helper functions.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| System registration at pipeline phase | `ecs_system_desc_t` + `ecs_entity_init` + `ecs_system_init` boilerplate | `CEL_System(Name, .phase = X) { cel_run { body } }` | CELS handles phase binding, entity creation, readonly suspension, and the two-pass registration pattern |
| Component registration | Raw `ecs_component_init` | `cels_ensure_component(&id, name, size, align)` | Already used correctly in ncurses_module.c |
| Phase entity lookup | `ecs_pair(EcsDependsOn, EcsPhase)` + `EcsPhase` array | `OnLoad`, `PreRender`, `PostRender` shorthand macros | CELS maps these to the correct flecs entities internally |

**Key insight:** All 3 system registrations follow the exact same pattern and can be directly replaced by `CEL_System` with `cel_run`. The only code that truly needs flecs.h is the observer registration (which has no CELS equivalent) and the `ecs_set_id` calls (which need a bridge function).

## Common Pitfalls

### Pitfall 1: Observer Callbacks Need ecs_iter_t Access

**What goes wrong:** The observer callbacks (`on_window_config_set`, `on_window_config_removed`) receive `ecs_iter_t*` and access `it->entities`, `it->count`, `it->world`, and `ecs_field()`. You cannot simply change the callback signature without changing how the observer is registered.

**Why it happens:** CEL_System's `cel_run` pattern works for systems (periodic callbacks) but NOT for observers (event-driven callbacks). Observers are a different ECS concept.

**How to avoid:** Keep the observer callbacks with their `ecs_iter_t*` signature but isolate them in the bridge file (which has `#include <flecs.h>`). Or, refactor the callbacks to use the bridge's wrapper types. The bridge file is the single point of flecs contact.

### Pitfall 2: CEL_System Generates File-Scope Statics

**What goes wrong:** `CEL_System(Name, ...)` expands to file-scope static variables and a `Name_register()` function. If you define `CEL_System(NCurses_InputSystem, .phase = OnLoad)` in `tui_input.c`, the generated `NCurses_InputSystem_register()` is NOT static -- it has external linkage. But the system body function IS static.

**Why it happens:** The macro is designed for cross-TU registration (module init calls `Name_register()`).

**How to avoid:** This is actually the desired behavior. The existing `ncurses_register_input_system()` function (which overrides the weak stub) can call `NCurses_InputSystem_register()` internally. The weak stub override pattern is preserved.

### Pitfall 3: draw_test Target Compiles Without CELS

**What goes wrong:** `draw_test` compiles `tui_frame.c` without `CELS_HAS_ECS` defined (it doesn't link cels). If ECS-specific code is not properly guarded, draw_test breaks.

**Why it happens:** `tui_frame.c` already uses `#ifdef CELS_HAS_ECS` guards. The `CEL_System` definitions and `ncurses_register_frame_systems()` are inside this guard.

**How to avoid:** Keep the `#ifdef CELS_HAS_ECS` guard in `tui_frame.c`. The `CEL_System` definitions MUST go inside this guard. Non-ECS functions (`tui_frame_init`, `tui_frame_begin`, `tui_frame_end`) stay outside the guard.

### Pitfall 4: Static g_window_entity Type Change

**What goes wrong:** `g_window_entity` is currently `ecs_entity_t`. Changing to `cels_entity_t` is a simple typedef change (both are `unsigned long long` / `uint64_t`), but if any code passes it to a function expecting `ecs_entity_t`, the bridge file needs to cast.

**Why it happens:** Both types are the same underlying integer type but semantically different.

**How to avoid:** Use `cels_entity_t` consistently in all non-bridge files. The bridge file accepts `cels_entity_t` parameters and casts to `ecs_entity_t` internally (or vice versa, since they're the same type).

### Pitfall 5: Registration Order in ncurses_register_input_system()

**What goes wrong:** `ncurses_register_input_system()` currently does ncurses setup (define_key, mousemask, running pointer) BEFORE registering the ECS system. With `CEL_System`, the system definition is at file scope. The `NCurses_InputSystem_register()` call must still come AFTER the ncurses setup within `ncurses_register_input_system()`.

**Why it happens:** `CEL_System` definition is file-scope, but registration is explicit via `Name_register()`.

**How to avoid:** In `ncurses_register_input_system()`, do all ncurses setup first, then call `NCurses_InputSystem_register()` last. This preserves the current ordering.

### Observer Bridge Pattern (Window)

The observer callbacks need a bridge file because CELS has no public observer API.

**Bridge file (`ncurses_ecs_bridge.c`):**
```c
#include <cels/cels.h>
#include <flecs.h>  /* ONLY file with this include */
#include "cels-ncurses/tui_ncurses.h"
#include "ncurses_ecs_bridge.h"

void ncurses_observer_register(const char* name,
                                cels_entity_t component_id,
                                ncurses_event_type_t event,
                                void (*callback)(ecs_iter_t*)) {
    ecs_world_t* world = cels_get_world(cels_get_context());
    ecs_observer_desc_t desc = {0};
    ecs_entity_desc_t entity_desc = {0};
    entity_desc.name = name;
    desc.entity = ecs_entity_init(world, &entity_desc);
    desc.query.terms[0].id = (ecs_id_t)component_id;
    desc.events[0] = (event == NCURSES_EVENT_ON_SET) ? EcsOnSet : EcsOnRemove;
    desc.callback = callback;
    ecs_observer_init(world, &desc);
}

void ncurses_component_set(cels_entity_t entity,
                            cels_entity_t component_id,
                            const void* data,
                            size_t size) {
    ecs_world_t* world = cels_get_world(cels_get_context());
    ecs_set_id(world, (ecs_entity_t)entity, (ecs_id_t)component_id, size, data);
}
```

**NOTE:** The observer callbacks still receive `ecs_iter_t*` because that is the flecs callback signature. This means `tui_window.c` still needs access to `ecs_iter_t` and `ecs_field`. Two options:

1. **Keep observer callbacks in bridge file** (moves ~60 lines from tui_window.c to bridge file)
2. **Forward-declare `ecs_iter_t` and use void* casts** (fragile, not recommended)

**Recommendation:** Option 1 -- move observer callbacks into the bridge file. This completely eliminates flecs.h from tui_window.c.

## Design Decision: Bridge File vs. ifdef Guards

Two approaches were considered:

### Option A: Bridge File (RECOMMENDED)

Create `src/ncurses_ecs_bridge.c` as the SOLE file with `#include <flecs.h>`. Move observer callbacks there. Other files call bridge functions.

**Pros:**
- Zero `#include <flecs.h>` in any other source file (success criterion #1 met cleanly)
- Single point of maintenance for ECS backend changes
- Observer callback code stays testable
- Clean module boundary

**Cons:**
- Observer callback code moves away from tui_window.c (slightly less cohesive)
- New file to add to CMakeLists.txt

### Option B: ifdef Guards

Keep observer code in tui_window.c with `#ifdef CELS_LACKS_OBSERVER_API` + `#include <flecs.h>`.

**Pros:**
- Observer code stays next to window code
- No new files

**Cons:**
- Violates success criterion #1 ("No cels-ncurses .c file includes `<flecs.h>` directly except behind feature guards")
- "Feature guard" is vague -- technically meets the letter but not the spirit
- Multiple files still depend on flecs.h
- Harder to track when CELS adds observer support

**Decision: Option A (bridge file)** -- it fully satisfies the success criteria and provides the cleanest module boundary.

## Open Questions

### 1. Should observer callbacks be refactored to use CELS types?

- **What we know:** The observer callbacks (`on_window_config_set`, `on_window_config_removed`) currently use `ecs_iter_t*` and `ecs_field()`. Moving them to the bridge file eliminates the flecs dependency from tui_window.c, but the callbacks themselves still use flecs types.
- **What's unclear:** Should the callbacks be rewritten to accept CELS-level parameters (entity ID array, component data pointer, count), making them purely CELS-typed? Or is it acceptable to keep them using flecs types since they live in the bridge file?
- **Recommendation:** Keep flecs types in the bridge file. The bridge IS the flecs boundary. Refactoring callbacks to use CELS types would require a trampoline function (more code, more indirection, no real benefit since the bridge file already contains flecs).

### 2. Will CELS add observer support in the future?

- **What we know:** CELS internally uses `ecs_observer_init()` in `composition_engine.c` and `world_manager.c`. The public API has `CEL_Condition` for lifecycle visibility (different concept). There is no `CEL_Observer` for event-driven callbacks in the current API.
- **What's unclear:** Whether CELS plans to add a public observer API. The planning docs mention observer-related design discussions.
- **Recommendation:** Design the bridge wrapper so it can be replaced with a CELS macro in the future. Name it `ncurses_observer_register()` (module-prefixed) so it doesn't conflict with any future `cels_observer_register()`.

### 3. Should ncurses_window_frame_update() move to bridge file?

- **What we know:** This function calls `ecs_set_id()` in 3 places to update WindowState. With the bridge wrapper, it would call `ncurses_component_set()` instead.
- **What's unclear:** The function also accesses `g_window_entity` (static to tui_window.c). Moving it to the bridge would require exposing g_window_entity or passing it as a parameter.
- **Recommendation:** Keep `ncurses_window_frame_update()` in tui_window.c. Replace `ecs_set_id()` calls with `ncurses_component_set()` from the bridge. This avoids exposing internal state.
